# 02 - 线路规划器算法设计

## 问题定义

给定 N 个站点（坐标、类别）和 O-D 流量模式，**一次性**输出一组线路，使得：

1. 所有关键 O-D 流向有直达或一次换乘路径
2. 线路数量不超过 max_lines (7)
3. 各类别站点覆盖率最大化
4. 运行一天 (1200 tick) 后的拥堵风险最小化

这是一个**组合优化问题**，搜索空间巨大（N 个站点分配到 ≤7 条线路，每条线路有站序）。传统精确算法不可行，需要用学习型方法。

**关键约束**：线路规划器只在规划阶段执行一次，规划完成后调用 `lock_lines()` 锁定线路，此后不可新建/延伸/插入线路。调度器只能在固定线路上做资源调度。

## 算法选择：Pointer Network + 强化学习

### 为什么选 Pointer Network

线路规划的本质是"从站点集合中选一个子集并排序"——这正是 Pointer Network 擅长的。Pointer Network 是 Seq2Seq 模型的变体，解码时每一步从输入序列中"指针"选择一个元素，天然适合组合优化。

### 为什么用 RL 而非监督学习

- 没有现成的"最优线路规划"标签数据
- 线路规划的好坏取决于下游调度和乘客行为，只能在环境中评估
- RL 可以直接优化游戏目标（减少等待、避免拥堵）

### 模型架构

```
输入: GNN 节点嵌入 (N, 64) + 全局特征 (145)
      │
      ▼
┌─────────────────────────────────────┐
│  线路规划器                           │
│                                      │
│  1. 编码阶段                          │
│     全局特征 → MLP(145 → 128)        │
│     节点嵌入 → LSTM encoder (64→128)  │
│                                      │
│  2. 解码阶段 (逐线生成)               │
│     对每条线路 k = 1..K:              │
│       初始状态: s_k = MLP(global)     │
│       重复：                          │
│         attention over nodes → α_i   │
│         选站: station = argmax/sample │
│         更新: s_k = LSTM(s_k, e_i)   │
│       直到选到 <END> 或达到最大长度    │
│                                      │
│  输出: K 条线路，每条是站点 ID 序列     │
└─────────────────────────────────────┘
```

### 注意力机制

解码时使用 **Bahdanau attention**，让模型关注与当前规划最相关的站点：
- 未被任何线路覆盖的站点应获得更高注意力权重
- 当前线路的首端站点类别附近的站点应获得更高权重（保证线路类别覆盖）

### 动作空间设计

线路规划器的动作不是一次性输出所有线路，而是**逐线生成**：

1. 模型决定"是否创建新线路"（二分类：创建 / 停止）
2. 如果创建，进入该线路的站点选择循环：
   - 每步选择一个站点（Pointer Network 指针）
   - 或选择 `<END>` 结束当前线路
3. 回到步骤 1

这种设计让模型可以灵活决定线路数量（0 到 max_lines）。

**规划完成后**：调用 `AIWorld.build_lines(line_definitions)` 创建线路，再调用 `place_initial_trains(train_placements)` 分配初始列车，最后 `lock_lines()` 锁定。此后进入运行阶段，线路不可更改。

### 训练方式：PPO (Proximal Policy Optimization)

**为什么选 PPO**：
- 比策略梯度更稳定（clip 机制防止更新过大）
- 比 DDPG/A2C 更易调参
- 在连续决策场景已被广泛验证

**奖励设计**——规划完成后在 AIWorld 中运行一天 (1200 tick)，用每日结算报告计算奖励：

| 奖励项 | 计算 | 权重 |
|--------|------|------|
| category_coverage | 结算报告中 overall_coverage_ratio | +5.0 |
| od_reachability | 有可达路径的 O-D 对占比 | +3.0 |
| avg_transfers | 平均换乘次数的负数 | -1.0 |
| line_balance | 各线路站数方差的负数 | -0.5 |
| passengers_arrived | 结算报告中 passengers_arrived_today / 预期客流 | +2.0 |
| max_waiting | 结算报告中 max_station_passengers / overcrowd_limit | -3.0 |
| survived | 结算报告中 survived (bool) | +10.0 / -10.0 |
| profit | 结算报告中 profit 的归一化值 | +1.0 |

**奖励的关键洞察**：线路规划的好坏必须放入环境中运行一整天才能评估。训练流程是：

```
规划线路 → 放初始列车 → lock_lines() → 运行 1200 tick (一天)
→ 读取每日结算报告 → 计算奖励 → 更新策略
```

与之前的设计不同，奖励来自一天结束后的完整结算，而非短时间模拟，这样评估更准确。

### 课程学习 (Curriculum Learning)

线路规划从简单场景开始训练，逐步增加难度：

| 阶段 | 站点数 | 线路数 | O-D 复杂度 | 天数 |
|------|--------|--------|-----------|------|
| 1 | 6-8 | 3 | 仅早高峰 O-D | 运行 1 天 |
| 2 | 10-12 | 4-5 | 早高峰 + 晚高峰 | 运行 1 天 |
| 3 | 14-16 | 5-6 | 全部时段 | 运行 1 天 |
| 4 | 18-20 | 7 | 全部时段 | 运行 1 天 |

注意：课程学习中无"动态站点"阶段，因为 AI 世界中站点在 setup 时一次性生成，不存在新增站点的情况。

## 备选方案对比

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| Pointer Network + RL | 直接输出生成，架构优雅 | 训练不稳定性 | **采用** |
| Graph Attention + 节点分类 | 简单 | 只能分线号无法排序 | 排除 |
| 遗传算法 | 不需训练 | 无法利用学习特征 | 后备方案 |
| 模拟退火 | 可并行 | 同上 | 后备方案 |
